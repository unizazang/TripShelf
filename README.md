
# TripShelf

TripShelf는 여행 기록을 남기고, 그 기록을 바탕으로 책 주문을 생성할 수 있는 콘텐츠 서비스입니다.

---

## 1. 서비스 소개

### 한 줄 소개
여행별로 사진과 글을 기록하고, 완성된 여행 기록을 책 주문으로 연결할 수 있는 웹 애플리케이션입니다.

### 타겟 사용자
- 여행 기록을 정리하고 싶은 사용자
- 여행 후 사진과 글을 한 곳에 모아두고 싶은 사용자
- 기록을 나중에 책 형태로 남기고 싶은 사용자

### 서비스 핵심 방향
이 서비스의 본체는 **여행 기록 콘텐츠 서비스**입니다.  
책 주문 기능은 여행 기록이 충분히 쌓인 뒤 사용하는 **부가 기능**입니다.

이번 과제에서는 “책 만들기 자체”보다  
**콘텐츠 생성 → 조회 → 누적 → 주문 연결** 흐름이 자연스럽게 작동하도록 만드는 것을 가장 중요하게 보았습니다.

## 3. 실행 방법 (Docker)

과제 요구사항에 맞게, 심사자가 README만 보고 바로 실행할 수 있도록 Docker Compose 기준으로 구성했습니다.
* 기본값이 compose.yaml에 포함되어 있어 별도 환경변수 설정 없이도 실행 가능합니다.
   필요하면 .env.example를 복사해 값을 변경해서 사용할 수 있습니다.

### 1) 저장소 클론
```bash
git clone <repo-url>
cd <repo>
````

### 2) 환경변수 준비

```bash
cp .env.example .env
```

### 3) 실행

```bash
docker compose up --build
```

### 4) 접속

```txt
http://localhost:3000
```

### 5) 포트 변경 방법

기본 포트는 3000입니다.
다른 포트를 사용하려면 `.env` 파일의 `APP_PORT` 값을 변경하면 됩니다.

예시:

```env
APP_PORT=3010
```

이 경우 접속 주소:

```txt
http://localhost:3010
```

### 6) 실행 직후 확인 가능한 더미 데이터

실행 직후 아래 더미 데이터가 자동으로 들어가 있습니다.

* 여행 9건
* 여행 기록 20건
* 주문 4건

즉, 로그인 없이도 바로 서비스 흐름을 확인할 수 있습니다.

## 5. 기술 스택 및 아키텍처

### 사용한 기술 스택

* Frontend: Next.js (App Router), TypeScript, Tailwind CSS
* Backend: Next.js Route Handler
* Database: PostgreSQL
* Infra: Docker, Docker Compose

### 왜 이 스택을 선택했는가

#### Next.js

페이지와 API 라우트를 한 프로젝트 안에서 함께 관리하기 쉬워서 선택했습니다.
App Router를 사용해 화면 구조와 서버 로직을 자연스럽게 연결할 수 있었습니다.

#### TypeScript

여행, 기록, 주문 데이터 구조를 명확하게 관리하기 위해 선택했습니다.
기능이 늘어날수록 타입 안정성이 도움이 되었습니다.

#### Tailwind CSS

짧은 과제 기간 안에 화면을 빠르게 만들고 정리하기 위해 선택했습니다.

#### PostgreSQL

처음에는 **Supabase**를 사용하려고 했습니다.
초기 구조도 Supabase를 기준으로 잡고, 로컬 개발도 그 방향으로 진행하려고 했습니다.

하지만 과제를 진행하면서 이 과제에서 더 중요한 것은
**심사자가 README만 보고 `docker compose up` 한 번으로 앱과 데이터를 바로 확인할 수 있어야 한다는 점**이라고 판단했습니다.

그래서 최종적으로는 **독립 실행성과 심사 재현성**을 더 높이기 위해 PostgreSQL 직결 구조로 변경했습니다.

즉, 이번 과제에서의 중요한 기술 선택 기준은
“스택 자체의 화려함”보다 **과제 요구사항에 더 잘 맞는 실행 구조인가**였습니다.

### 주요 아키텍처

* `app/`: App Router 기반 페이지와 API route
* `components/`: UI 컴포넌트
* `services/`: DB 조회/비즈니스 로직
* `lib/db.ts`: PostgreSQL 연결 관리
* `docker/postgres/init/`: 스키마 및 시드 데이터 초기화 스크립트

### 디렉터리 구조

```txt
src/
  app/
    page.tsx
    trips/
      new/page.tsx
      [id]/page.tsx
    orders/
      page.tsx
      [id]/page.tsx
    api/
      trips/route.ts
      entries/route.ts
      entries/[id]/route.ts
      orders/route.ts
      orders/[id]/route.ts
      export/[orderId]/route.ts

  components/
    trips/
    entries/
    orders/

  services/
    trips.ts
    entries.ts

```

## 7. 설계 의도

이 과제에서 가장 중요하게 본 것은
**“콘텐츠 서비스가 본체이고, 책 만들기는 그 위의 부가 기능이어야 한다”**는 점이었습니다.

그래서 먼저 여행 기록 서비스 자체가 자연스럽게 동작하도록 만들고,
그 이후에만 책 주문 기능을 붙였습니다.

즉, 구조를 다음 순서로 설계했습니다.

1. 여행 생성
2. 여행 기록 작성
3. 기록 조회 / 수정 / 삭제
4. 여행 콘텐츠 누적
5. 책 주문 생성
6. 주문 상태 관리
7. 주문 데이터 익스포트

이 흐름을 통해
“책 주문 앱”이 아니라 **여행 기록 콘텐츠 서비스**처럼 보이도록 하는 것을 목표로 했습니다.

## 8. 사업적 가능성

여행 기록은 단순 사진 저장보다
글과 날짜, 감정, 경험을 함께 남길 수 있다는 점에서 차별점이 있습니다.

또한 기록이 쌓일수록 이를 책 형태로 소장하고 싶어지는 수요가 생기기 때문에,
콘텐츠 서비스와 인쇄/제작 기능을 자연스럽게 연결할 수 있다고 생각했습니다.

즉, 사용자는 먼저 콘텐츠를 쌓고,
그 결과물이 충분히 가치 있다고 느껴졌을 때 책 주문 기능을 사용하게 됩니다.

## 10. AI 사용 중 겪은 실패 또는 문제

이번 과제에서 AI 사용 중 가장 크게 겪은 문제는
**처음에는 제가 익숙한 Supabase 기반 설계가 자연스럽다고 생각했지만, 과제 요구사항과 완전히 맞는 실행 구조인지는 다시 판단이 필요했다는 점**이었습니다.

* 로컬 개발이 편리함
* CRUD 구현 속도가 빠름
* 익숙한 방식으로 빠르게 시작 가능

하지만 진행하면서 다시 문서를 검토했을 때,
이 과제는 단순히 기능 구현만이 아니라
**심사자가 바로 실행 가능한 독립 서비스**여야 한다는 점이 더 중요하다고 판단했습니다.

그래서 중간에 방향을 바꿔,

* Supabase 기반 구조를 버리고
* PostgreSQL + Docker Compose 구조로 리팩터링했습니다.

이 과정에서 느낀 점은,
AI가 처음 제시한 방향이 빠르게 시작하는 데는 도움이 되었지만,
**최종 제출 요구사항에 정말 맞는지는 사람이 끝까지 판단해야 한다**는 것이었습니다.

즉, 이번 과제에서 가장 중요한 판단 중 하나는
**“개발 편의성보다 심사 재현성과 독립 실행성을 우선한다”**는 것이었습니다.
